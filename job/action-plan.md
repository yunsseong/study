## 현재 상태 (솔직한 진단)

| 항목 | 상태 | 평가 |
|------|------|------|
| **알고리즘** | 7일간 44문제, 매일 풀이 중 | 좋은 습관, 유지 필요 |
| **Spring Boot** | 기초 수준 (간단한 API 서버 가능) | **가장 큰 갭** |

### 프로젝트 1개에 집중 (4-5주)

2개를 얕게 하는 것보다 **1개를 제대로** 하는 게 낫다.

**추천: 선착순 쿠폰/이벤트 시스템**

이 주제를 추천하는 이유:
- 동시성 제어 → 빅테크 면접 핵심 주제
- Redis, Kafka → 실무 필수 기술
- 성능 테스트 → 숫자로 역량 증명 가능
- Spring Boot 심화 기능 자연스럽게 활용

**기술 스택**:
```
Java 21 + Spring Boot 3.x
Spring Data JPA + MySQL
Spring Security + JWT
Redis (캐싱 + 분산 락)
Apache Kafka (비동기 처리)
Docker + Docker Compose
GitHub Actions (CI/CD)
AWS EC2 + RDS (배포)
JUnit5 + Mockito (테스트)
```

**구현 순서** (주차별):
| 주차 | 목표 | 구현 내용 |
|------|------|---------|
| 1주 | 기본 API | 사용자 인증(JWT), 쿠폰 CRUD, 이벤트 CRUD |
| 2주 | 동시성 | 선착순 발급 (비관적 락 → Redis 분산 락으로 개선) |
| 3주 | 비동기 | Kafka 도입 (발급 요청 큐잉 → 비동기 처리) |
| 4주 | 배포 | Docker + AWS 배포 + GitHub Actions CI/CD |
| 5주 | 문서화 | README, 아키텍처 다이어그램, 성능 테스트, 블로그 |

**트러블슈팅 기록 예시** (면접 소재):
```
## 문제: 동시 1000건 요청 시 쿠폰 초과 발급

### 상황
100개 한정 쿠폰에 동시 1000건 요청 → 137개 발급됨 (37개 초과)

### 원인 분석
MySQL의 기본 격리수준(REPEATABLE READ)에서 재고 확인과 차감 사이에
다른 트랜잭션이 개입 → Race Condition 발생

### 해결 과정
1차: 비관적 락 (SELECT FOR UPDATE) → 정합성 확보, 하지만 TPS 120으로 하락
2차: Redis 분산 락 (Redisson) → 정합성 확보 + TPS 850 달성

### 배운 점
- 동시성 문제는 단일 서버에서도 발생한다
- DB 락은 정합성을 보장하지만 성능 병목이 된다
- Redis 분산 락이 성능과 정합성 모두 해결할 수 있다
```
